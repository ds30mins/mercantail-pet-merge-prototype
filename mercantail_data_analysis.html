2. Event breakdown by session

Engagement

Counts of each event type per session, derived from the raw events table using GROUP BY and conditional COUNT(CASE WHEN ...). This query reconstructs the session summary from raw events, validating the summary table figures.

Query 2 — Event breakdown 
-- Count each event type per session from raw event log
-- Uses conditional COUNT(CASE WHEN) to pivot event types into columns
SELECT
    session,
    COUNT(*) AS total_events,
    COUNT(CASE WHEN event_type = 'item_spawned'    THEN 1 END) AS spawned,
    COUNT(CASE WHEN event_type = 'item_merged'     THEN 1 END) AS merged,
    COUNT(CASE WHEN event_type = 'order_fulfilled' THEN 1 END) AS orders,
    COUNT(CASE WHEN event_type = 'shop_opened'     THEN 1 END) AS shop_opens,
    COUNT(CASE WHEN event_type = 'energy_refilled' THEN 1 END) AS energy_refills
FROM events
GROUP BY session
ORDER BY session;

3. Player funnel analysis

Funnel

Funnel built from scratch using UNION ALL to stack separate stage queries into one result. Tier figures use the video-verified summary table — raw event log tier data is incomplete for Basic Data sessions (S01–S07, S09) where the embedded logger was not active.

Query 3 — Player funnel (UNION ALL) 
-- Build funnel from raw event log using UNION ALL
-- Each block is one funnel stage; COUNT(DISTINCT session) prevents double-counting
-- Tier stages use summary table (video-verified) — see footnote
SELECT 'Started session' AS funnel_stage, 1 AS stage_order,
       COUNT(DISTINCT session) AS sessions, 10 AS total,
       ROUND(COUNT(DISTINCT session) * 100.0 / 10, 0) AS pct
FROM events
UNION ALL
SELECT 'Fulfilled ≥1 order', 2,
       COUNT(DISTINCT session), 10,
       ROUND(COUNT(DISTINCT session) * 100.0 / 10, 0)
FROM events WHERE event_type = 'order_fulfilled'
UNION ALL
SELECT 'Opened shop', 3,
       COUNT(DISTINCT session), 10,
       ROUND(COUNT(DISTINCT session) * 100.0 / 10, 0)
FROM events WHERE event_type = 'shop_opened'
UNION ALL
SELECT 'Used energy refill', 4,
       COUNT(DISTINCT session), 10,
       ROUND(COUNT(DISTINCT session) * 100.0 / 10, 0)
FROM events WHERE event_type = 'energy_refilled'
UNION ALL
SELECT 'Reached Tier 2+', 5,
       COUNT(DISTINCT session), 10,
       ROUND(COUNT(DISTINCT session) * 100.0 / 10, 0)
FROM summary WHERE max_tier_reached >= 2  -- video-verified
UNION ALL
SELECT 'Session >3 minutes', 6,
       COUNT(DISTINCT session), 10,
       ROUND(COUNT(DISTINCT session) * 100.0 / 10, 0)
FROM summary WHERE session_length_min > 3
UNION ALL
SELECT 'Reached Tier 5', 7,
       COUNT(DISTINCT session), 10,
       ROUND(COUNT(DISTINCT session) * 100.0 / 10, 0)
FROM summary WHERE max_tier_reached = 5  -- video-verified
ORDER BY stage_order;

4. Gold economy — P08 deep dive

Economy

Time-series analysis of gold accumulation across P08's two sessions. Only possible with Rich Data from the embedded logger — Basic Data sessions did not capture continuous economy state. Query filters to P08 and tracks gold_after at each event timestamp.

Query 4 — Gold economy over time (P08) 
-- Track gold balance over time for P08 (Rich Data sessions only)
-- gold_after - gold_before shows earn vs spend events
-- Saved as table: CREATE TABLE gold_economy_p08 AS ...
SELECT
    session,
    timestamp,
    event_type,
    gold_before,
    gold_after,
    gold_after - gold_before AS gold_change
FROM events
WHERE player = 'P08'
AND event_type != 'session_ended'
ORDER BY session, timestamp;

6. Session quality classification

Session quality

Uses a JOIN between the summary and events tables combined with CASE WHEN classification to assign each session a quality tier. The subquery pre-aggregates events per session; the outer query joins it to summary metrics.

Query 6 — Session quality (JOIN + CASE WHEN) 
-- Join summary and events tables to classify session quality
-- CASE WHEN applies industry benchmark thresholds (GameAnalytics 2025)
-- Subquery aggregates events per session before joining
SELECT
    s.session, s.player,
    s.session_length_min, s.orders_fulfilled,
    s.merge_ratio, s.max_tier_reached,
    e.total_events, e.shop_opens, e.energy_refills,
    CASE
        WHEN s.session_length_min < 1   THEN 'Early Churn'
        WHEN s.session_length_min < 3.5 THEN 'Short Session'
        WHEN s.session_length_min < 10  THEN 'Engaged'
        ELSE 'Power User'
    END AS session_quality
FROM summary s
JOIN (
    SELECT session,
           COUNT(*) AS total_events,
           COUNT(CASE WHEN event_type = 'shop_opened'     THEN 1 END) AS shop_opens,
           COUNT(CASE WHEN event_type = 'energy_refilled' THEN 1 END) AS energy_refills
    FROM events GROUP BY session
) e ON s.session = e.session
ORDER BY s.session_length_min DESC;

Funnel finding: T1→T3 is strong. T4→T5 collapses — and there are no T5 orders.

Players merge efficiently at T1–T3 (96% and 89% of theoretical baseline). The drop begins at T4 (78%) and collapses at T5: only 21 T5 items produced against an expected 59 — 36% efficiency. This is rational player behavior, not random attrition. With no T5 orders to justify the ~50-spawn investment required per T5 item, players consume T4 items for lower-tier orders instead. P08 flagged this directly and independently: "Why no Lv5 order? What's the purpose of Lv5 item?"

Next action: Introduce at least one rotating T5 order with a Gold reward (~150–200 Gold) that justifies the merge chain cost. Closes the motivation dead end and creates a natural premium spend moment.
